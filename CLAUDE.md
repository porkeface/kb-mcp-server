# 全局规范

## 称呼规范

- 每次回复必须称呼用户为 **老大**

## Python 环境管理

- 使用 **uv** 管理 Python 依赖和虚拟环境
- 安装依赖：`uv add <package>`
- 运行命令：`uv run <command>`
- 运行测试：`uv run pytest tests/ -q`（需设置 `PYTHONPATH=.`）
- 同步依赖：`uv sync`
- 锁文件：`uv.lock`（必须提交）

## UI 开发规范

- **写 UI 时必须调用对应的 skill**，不要直接手写
- 使用 `/web-design` 或相关 skill 进行前端设计
- 使用 `/component` 或相关 skill 生成组件代码

## 团队协作模式

采用 **team 模式** 开发：

1. **及时讨论** — 团队成员遇到问题要及时讨论，不要闷头做
2. **产品经理监察** — 产品经理要时刻关注工程进度和质量
3. **分工明确** — 前端、后端、测试、架构各有专责

## MCP 工具使用

- **审计与测试时优先调用 MCP 工具**（如浏览器mcp、代码搜索、文件分析、API 测试等），提高效率
- 可用的 MCP 工具包括但不限于：代码搜索、Web 搜索、文件系统操作、数据库查询等
- 遇到需要外部验证的逻辑，主动使用 MCP 工具获取真实数据而非假设

## 代码审计规范

代码完成后必须审计，采用 **递进式审计**：

1. **第一次审计** — 全面审查代码质量、逻辑正确性、安全性
2. **如果第一次审计发现问题** → 修复后进行 **第二次审计**
3. **如果第二次审计仍有问题** → 修复后进行 **第三次审计**
4. **循环执行** — 直到审计零问题为止

### 审计维度

- [ ] 代码逻辑是否正确
- [ ] 是否有未处理的边界情况
- [ ] 是否有硬编码的密钥或敏感信息
- [ ] 错误处理是否完善
- [ ] 测试覆盖是否充分（目标 80%+）
- [ ] 是否有并发安全问题

## 测试规范

- 所有测试必须 100% 通过才能提交
- 新功能必须有对应测试

## 提交规范

- 格式：`<type>: <description>`
- 类型：feat, fix, refactor, docs, test, chore, perf, ci
